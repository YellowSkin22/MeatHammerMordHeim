# Design: Adopt Raw Uncle-Mel Data Structure

**Date:** 2026-04-03
**Status:** Approved

## Summary

Replace all transformed/merged data files with verbatim copies of Uncle-Mel/JSON-derulo files,
stored in the same directory structure as the source repo. Update the full chain — sync script,
DataService, RosterModel, UI — to work with the raw format natively. No adapter layer.

---

## Section 1: Sync Script

The sync script becomes a pure file downloader. All transformation logic is removed.

**Behaviour:**
1. Lists subdirectories of `data/warbandFiles/` in the source repo via GitHub API — dynamically discovers all grade folders (`1a/`, `1b/`, `1c/`, and any future grades like `2a/`) without hardcoding
2. Downloads each warband file verbatim to `data/warbandFiles/{grade}/{file}.json`
3. Downloads `data/equipment.json`, `data/skills.json`, `data/magic.json`, `data/hiredSwords.json`, `data/maxStats.json` verbatim
4. Generates `data/warbandFiles/index.json` — the only generated file; a flat array of `{ grade, path }` entries listing every warband file. Needed because the app is a static site that cannot enumerate directories at runtime
5. Deletes the old `data/warbands.json` and `data/mergedEquipment.json` on first run
6. SHA-based change detection is kept so unchanged files are skipped on subsequent syncs
7. Hand-maintained files (`injuries.json`, `advancement.json`, `special_rules.json`) are never touched

**Result:** script goes from ~700 lines to ~150 lines. All transformation maps (`WARBAND_SPECIAL_SKILL_CATEGORIES`, `HIRED_SWORD_WARBAND_NAME_MAP`, `SPECIAL_SKILL_CATEGORY_MAP`, etc.) are removed.

## Section 2: Data Files

### Removed
- `data/warbands.json` — replaced by individual warband files
- `data/mergedEquipment.json` — replaced by `data/equipment.json`

### Downloaded verbatim from Uncle-Mel
| File | Notes |
|------|-------|
| `data/warbandFiles/{grade}/{warband}.json` | All grades, discovered dynamically |
| `data/warbandFiles/index.json` | Generated by sync — not in Uncle-Mel |
| `data/equipment.json` | Flat array; same content as current `mergedEquipment.json`, renamed |
| `data/skills.json` | Raw flat array with `subtype` field; replaces our categorised format |
| `data/magic.json` | Same filename, verbatim Uncle-Mel format |
| `data/hiredSwords.json` | Raw keyed object; replaces our transformed array |
| `data/maxStats.json` | New — race-based max stats; replaces `maxStats` block in `advancement.json` |

### Hand-maintained (format unchanged)
| File | Change |
|------|--------|
| `data/injuries.json` | Untouched |
| `data/advancement.json` | `maxStats` block removed; rest stays |
| `data/special_rules.json` | Untouched |

**Note on maxStats:** Uncle-Mel's `maxStats.json` is race-based (human, elf, dwarf, ogre, etc.)
with per-race, per-warband overrides and lowercase stat keys. This is richer and more rules-correct
than our current flat generic block.

## Section 3: DataService

### Loading
`DataService.loadAll()` now:
1. Fetches `data/warbandFiles/index.json`
2. Fires one `fetch()` per warband file in parallel; stores results in `DataService.warbandFiles[]`
3. Fetches all other data files in parallel; stores raw on DataService properties
4. Removes all pre-computation (`_buildSpellAccess()` etc.) — data is stored raw

### Public API changes

| Method | Change |
|--------|--------|
| `getWarband(id)` | Returns raw Uncle-Mel warband file object |
| `getWarband(id, subfaction)` | Subfaction variant — resolves which `skillAccess` entry applies for that subfaction |
| `getAllWarbands()` | Returns subfaction-expanded entries for the warband picker (one entry per subfaction where applicable) |
| `getEquipmentItem(id)` | Returns raw Uncle-Mel equipment item (id = slugified name) |
| `getEquipmentByCategory(type)` | Filters by Uncle-Mel `type` field directly (`"melee"`, `"missile"`, etc.) |
| `canWarbandAccess(item, warbandName)` | Unchanged logic — `permittedWarbands` already present in raw format |
| `getSkill(id)` | Searches flat skills array by slugified name |
| `getSkillsByCategory(subtype)` | Filters flat skills array by `subtype` string (e.g. `"Combat Skill"`) |
| `getSpell(id)` | Returns raw spell from `magic.json` spellLists |
| `getSpellsByList(listId)` | Returns raw spells for a spell list |
| `getMaxStat(statKey, race)` | Takes race parameter; looks up `maxStats.json` by race; returns value for lowercase stat key |
| `getSpellAccess(warbandName, fighterName)` | Extracted from old `_findSpellAccess` — cross-refs `magic.json` permittedWarbands |
| `getHiredSwordTemplate(type)` | Looks up by key in raw keyed object |
| `getAvailableHiredSwords(warbandName)` | Matches `permittedWarbands` display names against loaded warband `name` fields directly — no ID mapping needed |
| `getExpThreshold(count)` | Unchanged — reads from `advancement.json` |
| `getSpecialRuleDescription(name)` | Unchanged |

## Section 4: Warrior Object Shape (localStorage)

Stat keys switch to **lowercase** throughout the warrior object to match Uncle-Mel's format.

### Changed fields

| Before | After | Source field |
|--------|-------|-------------|
| `stats: { M, WS, BS, S, T, W, I, A, Ld }` | `stats: { m, ws, bs, s, t, w, i, a, ld }` | `fighter.statblock` |
| `baseStats: { M, WS, … }` | `baseStats: { m, ws, … }` | Same |
| `cost` from `template.cost` | `cost` from `fighter.costGc` | Uncle-Mel field rename |
| `startingExp` from `template.startingExp` | `startingExp` from `fighter.startingXp` | Uncle-Mel field rename |

### New field
- `race` — copied from `warbandFile.race` at warrior creation time. Required for race-aware `getMaxStat()` lookups.

### Unchanged fields
`id`, `type`, `typeName`, `name`, `isHero`, `equipment`, `skills`, `spells`, `injuries`,
`experience`, `advancementCount`, `missNextGame`, `specialRules`, `groupSize`,
`isHiredSword`, `isCustom`, `isPromotedHenchman`, `skillAccess`

### Backward compatibility
`Storage.loadAllRosters()` runs a one-time silent migration on load:
- If `warrior.stats` has any uppercase key → convert all stat keys to lowercase, re-save
- If `warrior.skillAccess` contains short IDs (`combat`, `shooting`) → convert to Uncle-Mel subtype strings (`"Combat Skill"`, `"Shooting Skill"`)

## Section 5: RosterModel

### `createWarrior(fighter, warbandFile, subfaction)`
- Signature changes to take raw Uncle-Mel fighter object + warband file
- `fighter.statblock` (lowercase) → `stats` and `baseStats`
- `fighter.costGc` → `cost`
- `fighter.startingXp` → `startingExp`
- `fighter.specialRules[].rulename` → `specialRules[]`
- `skillAccess` resolved from subfaction-aware `fighter.skillAccess` array
- `race` set from `warbandFile.race`
- `spellAccess` computed at creation via `DataService.getSpellAccess(warbandName, fighterName)`

### `createHiredSword(template)`
- Takes raw Uncle-Mel hired sword entry (value from keyed object)
- `template.statblock` (lowercase) → `stats` / `baseStats`
- `template.permittedWarbands` stays as display names

### `modifyStat(warrior, statKey, delta)`
- `statKey` is always lowercase (`'m'`, `'ws'`, etc.)
- Max lookup: `DataService.getMaxStat(statKey, warrior.race)`

### Unchanged
- `promoteHenchmanToHero()` — logic unchanged; stat keys are already lowercase on the warrior
- `calculateWarbandRating()` — no stat key references
- `getStatModified(warrior, statKey)` — same comparison, lowercase keys

## Section 6: UI

### Stat key references
All inline stat key references switch from uppercase to lowercase. Affected areas:
- Warrior card stat display
- Stat edit inputs
- Modified-stat highlight check (`stats[key] !== baseStats[key]`)
- Progress tab advancement roll application

A global search-and-replace handles most cases; any dynamic stat key construction is reviewed manually.

### Warband picker
`renderWarbandList()` calls `DataService.getAllWarbands()` which returns subfaction-expanded entries.
`id` and `name` fields remain present — dropdown rendering is unchanged.

### Warrior add dropdowns
`addWarriorFromSelect()` passes the raw Uncle-Mel fighter object to `RosterModel.createWarrior()`.
Fighter is fetched as `warbandFile.fighters.find(f => f.id === type)`.

### Equipment modal
Allowed equipment is resolved on-the-fly:
1. Look up `fighter.equipmentLists[]` references
2. Find matching lists in `warbandFile.equipmentLists[]`
3. Resolve item names against DataService equipment array

Modal display logic is unchanged; only data fetching changes. Hired swords use `equipmentAccess` category arrays from the raw hired sword entry.

### Skill modal
`warrior.skillAccess[]` stores Uncle-Mel subtype strings (`"Combat Skill"`, etc.).
Modal filters the flat skills array: `skills.filter(s => warrior.skillAccess.includes(s.subtype))`.

### Spell modal
`hasSpellAccess()` checks `warrior.spellAccess[]` (computed at creation, stored on warrior).
No change in logic.

### Unchanged
- Injury modal — reads hand-maintained `injuries.json`
- Progress tab advancement roll table — reads `advancement.json` (unchanged structure)
- PDF export — only stat key casing changes
- `UI.esc()` / `UI.escAttr()` — Uncle-Mel HTML in `lore`/`flavour` fields is never inserted raw

## Section 7: Error Handling & Migration

### Warband file loading
- Failure on `index.json` fetch → fatal error surfaced to user (no warbands can load)
- Failure on individual warband file → logged as warning, warband excluded from picker, app continues

### localStorage migration (one-time, silent)
On `Storage.loadAllRosters()`:
1. Inspect each warrior's `stats` object
2. If uppercase keys detected → convert `stats` and `baseStats` to lowercase, re-save roster
3. If `skillAccess` contains short IDs → convert to Uncle-Mel subtype strings

### Hired sword warband matching
`getAvailableHiredSwords(warbandName)` matches Uncle-Mel `permittedWarbands` display names
against loaded warband file `name` fields directly. The `HIRED_SWORD_WARBAND_NAME_MAP` lookup
table is removed from the codebase entirely.

---

## Implementation Order

1. Sync script (enables testing with real data immediately)
2. DataService (foundation all other changes depend on)
3. Storage migration (protects existing data before UI touches it)
4. RosterModel (warrior creation with new shapes)
5. UI (widest surface, last so it has correct data to render)
